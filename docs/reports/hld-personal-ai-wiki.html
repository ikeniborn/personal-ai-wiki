HLD — Personal AI Wiki

Веб-сервис · self-hosted, один инстанс на команду · сформировано 2026-06-21 · референс: ikeniborn/obsidian-ai-wiki

Командное веб-приложение, которое автоматически строит и поддерживает базу знаний (вики) из загруженных источников при помощи локального AI-харнесса. Переиспользует логику формирования вики из Obsidian-плагина (Ingest / Query / Lint+Fix / Init / Format над доменом) и отдаёт её через веб-UI, REST-API управления и удалённый MCP-сервер.

Область документа. Только HLD — контекст системы, container-вид, логические компоненты, потоки данных, модель данных и сквозные аспекты. Внутренности модулей, сигнатуры, схемы и шаблоны промптов переносятся в LLD.

1 · Цели и не-цели

Цели

  • Загрузка разнородных источников и надёжное хранение (PostgreSQL; объектное хранилище S3 — опционально).
  • Формирование и поддержка вики через AI-харнесс: Ingest, Query, Lint+Fix, Init, Format, Chat.
  • Хранение статей вики, графа связей между статьями и векторов по чанкам.
  • Ответы на вопросы по вики (RAG: гибрид вектор+FTS + BFS по графу) из UI и через MCP.
  • Просмотр и навигация: листинг статей, full-text поиск, граф связей, индекс домена.
  • Полноценный REST-API для загрузки, формирования, сохранения файлов и управления вики.
  • Многопользовательский доступ команды с ролями; работа офлайн на локальной LLM.

Не-цели (на этой фазе)

  • Multi-tenant SaaS / биллинг / изоляция организаций.
  • Облачная LLM как основной бэкенд (основной — локальный Ollama).
  • Совместное редактирование статей в реальном времени (CRDT).
  • Нативные мобильные клиенты.
  • Горизонтальное автомасштабирование воркеров (предполагается один инстанс).

3 · Рассмотренные варианты архитектуры

ВариантФормаКомпромиссВердикт
A · Модульный монолит + воркер выбран Один кодбейс FastAPI с чёткими модулями; воркер arq — второй процесс на том же коде; MCP смонтирован как под-приложение. Минимум ops для одного инстанса; модули всё равно тестируются изолированно; один артефакт деплоя. Выбран
B · Раздельные сервисы Отдельные деплои: API, воркер, MCP. Масштабируются независимо; больше движущихся частей, больше ops, сетевые хопы. Отложен — вернуться при росте нагрузки
C · Микросервисы по аспектам storage / harness / graph / vector — каждый свой сервис. Максимум гибкости; большой операционный оверхед и латентность. Отклонён — оверкилл для команды
Выбран A. Границы внутренних модулей (storage, harness, graph, vector, ingest, mcp, api) держим чистыми — так переход к B становится изменением деплоя, а не переписыванием.

4 · Container / context-вид

C4 container-вид: один инстанс, его внутренние контейнеры и внешние системы, с которыми он общается.

API/UI и MCP переиспользуют одни и те же модули поиска (vector + graph). Полный харнесс-синтез ответа живёт на стороне API/UI; MCP отдаёт сырой контекст (search_wiki) и оставляет рассуждение клиенту-агенту — это тонкий транспорт над общими модулями, а не параллельная реализация.

Абстракция провайдера харнесса

Харнесс зависит от интерфейса AgentProvider, а не от конкретного клиента. Фаза 1 поставляет единственный OpenAICompatProvider (Ollama / любой OpenAI-совместимый endpoint). Этот шов оставляет будущий провайдер (Claude и др.) аддитивным изменением.

Тела статей/источников лежат в PostgreSQL (Large Objects / bytea) через шов StorageBackend; PostgreSQL также держит метаданные, граф и векторы. S3/MinIO остаётся подключаемым позже без смены модели. Граф — это обычные реляционные рёбра, обходимые recursive CTE — без отдельного графового движка.

Жизненный цикл контента и происхождение

  • Детекция изменений: re-ingest сверяет checksum источника — неизменённое пропускается.
  • Каскад: удаление источника → чистка/пересборка осиротевших чанков, статей и битых рёбер графа.
  • Версия эмбеддингов: embedding_version на чанке; смена модели запускает фоновый reindex; поиск использует только текущую версию.
  • Provenance: статьи ссылаются на источники (citations) — основа доверия и работы Lint.
  • Версионирование: AI-правки публикуются сразу, полная история ревизий + откат; конкурентные правки — optimistic lock по current_rev.

12 · Сквозные аспекты

Безопасность

  • Хешированные пароли; подписанные session-cookie; хешированные API-ключи.
  • RBAC на каждом вызове API/MCP; viewer = только чтение.
  • Тела файлов в Postgres под общими транзакциями и RBAC; по-доменные ключи хранилища.
  • Скачивание URL за allowlist / SSRF-защитой.
  • Prompt injection: контент источников = данные, не инструкции; write-scope харнесса ограничен своим доменом; деструктивных tools нет; лимит шагов.
  • Санитайзинг markdown→HTML (allowlist) при рендере — защита от XSS.
  • CSRF-токены для всех мутирующих HTMX-форм (сессии).
  • Валидация загрузок: content-type sniffing, лимит размера, защита от zip-бомб (docx/epub).
  • Секреты (session-secret, креды БД, ключ LLM) — через env/secrets, не в коде/образе.

Наблюдаемость

  • Структурные логи; пошаговый лог задачи (JSONL) как agent-лог плагина.
  • SSE выводит живые шаги харнесса в UI.
  • Health-эндпоинты для api / worker / зависимостей.
  • Audit log: кто запустил ingest/lint/reindex, кто правил статью.

Надёжность

  • Задачи идемпотентны по checksum источника; безопасный повторный ingest.
  • Ретраи arq с backoff; упавшие задачи сохраняют свой лог.
  • Миграции через Alembic.
  • Бэкап/восстановление: pg_dump (метаданные + граф + векторы + тела LO); DR-план с учётом размера.
  • Отмена и таймаут зависших задач; graceful shutdown воркера на середине job.

Производительность

  • Векторный индекс (HNSW/IVFFlat), настроенный под размер корпуса.
  • BFS ограничен глубиной + hub threshold.
  • Эмбеддинги батчатся во время ingest.
  • Ollama держит одну модель за раз → LLM-задачи сериализуются (очередь по модели).

14 · Следующий шаг

После одобрения этого HLD переходим к LLD: интерфейсы и сигнатуры модулей, схема БД + миграции, определения промптов/инструментов для каждой операции харнесса, контракт AgentProvider, схемы запросов/ответов API, схемы MCP-инструментов и топология Docker Compose.

Отдельные пункты повестки LLD: принцип чанкования и подготовки векторов (гранулярность статья/заголовок/сущность, структура индекса по статьям домена + по сущностям); стратегия гибридного слияния (RRF/веса) и reranking; каскадное удаление и soft-delete; rate limiting, idempotency-ключи, bulk-upload; first-admin bootstrap; resumable ingest. Вне области первой версии: per-domain ACL, hot-reload конфига, webhooks, eval-харнесс качества.
